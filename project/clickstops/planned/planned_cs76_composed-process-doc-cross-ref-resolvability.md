## Goal

Make every cross-reference and harness-internal link in the **consumer-shipped composed process-doc bases** (`template/composed/OPERATIONS.md`, `REVIEWS.md`, `CONVENTIONS.md`) resolve against a surface the consumer is **guaranteed** to have — closing issue **#229** (dangling `INSTRUCTIONS.md` / `.github/copilot-instructions.md` sibling refs, plus the **B** label-wording and **C** path-precision nits) **and** CS81's deferred **R3** gap (dangling `docs/adr/*` process-base links) — and **extend** CS81's existing resolvability guard so neither regression can recur. This is the second consumer-doc invariant LRN-170 names, distinct from CS72's anchor-genericity guard.

## Background

CS72 made the 5 consumer *onboarding* docs anchor-generic (no `LRN-NNN` / `CSNN` / slug tokens) and guarded it with `scripts/check-consumer-template-genericity.mjs`. It deliberately scoped OUT the composed *process-doc* bases (C72-3) and a second failure mode (LRN-170): a consumer-shipped composed base can cross-reference a **sibling file the consumer may not sync**, producing a dangling ref in the rendered consumer doc. Such a ref carries no banned anchor token, so CS72's guard passes it.

Issue #229 (filed from `henrik-me/sub-invaders`'s v0.7.0 pin-bump PR #85) documents the concrete breakage. Line numbers below are current at HEAD `67f3b5d` (the R5 plan review ran at `0a4eff8`; CS83 shifted OPERATIONS.md's later-half refs by +1 without altering the cited content):

- **A (primary) — broken cross-refs to non-synced files.** `template/composed/OPERATIONS.md` and `REVIEWS.md` point at `INSTRUCTIONS.md` / `.github/copilot-instructions.md` as canonical sources. The unqualified sibling **cross-refs** are `template/composed/OPERATIONS.md:240, :591, :694, :948, :2304, :2390` and the § 2.6a/2.6c F3 "Source of truth" table cells `template/composed/REVIEWS.md:226, :333`. Legitimate *descriptive* mentions that must be **spared** (allowlisted) are the genericity-invariant doc list at `OPERATIONS.md:1992-2004` and the nonexistent-path *example* at `REVIEWS.md:249`. A consumer whose `harness.config.json` omits those files renders dangling references it **cannot fix locally** (composed sections are regenerated verbatim by `harness sync`, so a local edit fails `harness-sync-check`). Remediation is **audit-driven over every occurrence**, not limited to #229's originally-cited (now-drifted) line numbers.
- **B — inaccurate doctrine wording.** `template/composed/OPERATIONS.md:626` heads the issue-creation procedure "non-mutating to consumer labels", but its "Label preflight" step (`:648` onward) *creates* the `harness-orchestrator` label when missing. Rephrase to "non-destructive / non-overwriting (no `--force`)".
- **C — path-precision nit.** `template/composed/REVIEWS.md:592` uses bare `pr-evidence-lint.yml` where the full `template/managed/.github/workflows/pr-evidence-lint.yml` form is used elsewhere in the same file (`:603`).

**CS81 shipped the guard this CS extends.** After #229 was filed, CS81 (v0.11.0) added `scripts/check-doc-xref-resolvability.mjs`, which validates three doc-xref classes but **deliberately excludes the composed process-doc bases** (`OPERATIONS.md` / `REVIEWS.md` / `CONVENTIONS.md`) from its relative-link class — its docstring defers them as its own "R3 follow-up" because they carry pervasive `docs/adr/*` links. This CS is that follow-up: it **extends** CS81's guard (rather than adding a near-identically-named second script) to cover the process bases for both the not-guaranteed-sibling class (#229-A) and the deferred `docs/adr` class.

**Folded-in class — dangling `docs/adr/*` process-base links (CS81-R3).** At HEAD `67f3b5d` the composed process-base **core** carries harness-internal `docs/adr` references that ship under no consumer delivery surface (`docs/adr/` is in no consumer's `composed.files`/`managed.files`): `OPERATIONS.md:949, :1565, :1584, :1729, :1811, :2466` (relative markdown links, inline-code relative paths, and two hardcoded-`henrik-me/agent-harness`-slug `https://github.com/…` absolute URLs) and `CONVENTIONS.md:123, :131, :144` (incl. a convention-illustrating example at `:123`). `REVIEWS.md` has none. Because the composed **core** renders to BOTH the consumer doc and the harness self-host root, genericizing these also strips them from the harness's own root docs — an accepted consequence (C76-8, Risks R3). CS81 set the precedent: it genericized `READMEGUIDE.md`'s `docs/adr` links (ADR pointers dropped, explanatory prose kept).

**Relevance note (OBE for the original reporter).** Verified read-only this session: `henrik-me/sub-invaders`'s current `harness.config.json` now syncs **both** `INSTRUCTIONS.md` and `.github/copilot-instructions.md` as composed files (the CS72 managed→composed reclassification + a later pin), so #229-A no longer dangles for the original reporter. #229-A's value is now the **regression guard** plus the latent invariant for any consumer that omits those composed files (they are consumer-selectable, C76-2). #229-B and #229-C remain live in every consumer that syncs `OPERATIONS.md` / `REVIEWS.md`.

Because the repo self-hosts, every composed-base edit propagates to the rendered repo-root `OPERATIONS.md` / `REVIEWS.md` / `CONVENTIONS.md` via `harness sync` (templating placeholders such as `{{agent_suffix}}` are substituted in the roots, so roots are **not** byte-identical to the source); the `check-composed-blocks` lockstep lint + `harness sync --mode=check` enforce that the managed-core sections agree **after rendering**. The C76-8 `docs/adr` edits live in this managed core, so they land in both the source and the rendered roots.

## Risks + open questions

- **R1 — false positives on descriptive mentions (mitigated by C76-3).** Composed bases legitimately *describe* the onboarding-doc set (e.g. `OPERATIONS.md:1991-2003`; `REVIEWS.md:249`; `CONVENTIONS.md:18, :140`). The occurrence + commented-allowlist rule handles this: every occurrence must be qualified or explicitly allowlisted, so descriptive prose is permitted only via a justified allowlist entry — no fuzzy "is this a cross-ref?" heuristic. Residual risk: the allowlist must be re-audited when the bases change — covered by the guard failing on any *new* unqualified occurrence.
- **R2 — "guaranteed-present" is consumer-config-dependent.** The not-guaranteed set is an explicit assumption (C76-2), not derivable from any single consumer's config; if a future change makes `INSTRUCTIONS.md` universally delivered, the set must be updated. Encode it as documented config, not a magic constant.
- **R3 — self-host impact of `docs/adr` genericization (C76-8).** Dropping inline ADR links from the composed core also strips them from the harness's OWN root `OPERATIONS.md` / `CONVENTIONS.md`. Accepted consequence (the ADRs remain under `docs/adr/`); the claimer must confirm before the irreversible self-host edits, and may instead migrate the hotlinks to the harness local block if the inline links are deemed worth preserving self-host-side.
- **R4 — sequencing vs paused CS65.** CS65 (⏸️ Paused, owner omni-ah-c2, branch `cs65/content`) also rewrites the root `OPERATIONS.md` (right-sizing / extraction). CS76's `OPERATIONS.md` edits may need a rebase if CS65 resumes and lands first. CS65 is currently blocked on user gates, so CS76 may proceed; coordinate at claim time.
- **R5 — guard-target scope (was OQ2).** The extended guard targets the `template/composed/` sources; CS81's existing classes (a)/(b) already scan the rendered roots, so root coverage for the new class is optional defence-in-depth.
- **(Resolved, C76-3/C76-7)** The guard is an **extension** of CS81's `check-doc-xref-resolvability.mjs` (not a new script), which drives the **Patch** SemVer and avoids a second near-identically-named guard.
