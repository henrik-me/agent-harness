## Goal

Make the repo's existing "Knowledge lives in the repo, not agent memory" doctrine (INSTRUCTIONS.md §45–61) both **mechanically enforced** and **proactively surfaced**, so no agent (orchestrator or sub-agent) can silently persist durable knowledge into the Copilot memory feature instead of `LEARNINGS.md` / the process docs. Scope is the **self-host only** (a committed repo-level hook + a self-host local-block instruction line); shipping the hook to consumer repos via a `template/` copy is explicitly out of scope.

## Background

- The doctrine is currently **doctrine-only**: it lives in `INSTRUCTIONS.md` §45–61, a file agents read on demand — not every turn. The **always-loaded** custom-instructions file `.github/copilot-instructions.md` (injected into every session) says nothing about memory, and there is **no `.github/hooks/`** directory in the repo (verified this session: `Test-Path .github/hooks` → False). So the only "trigger" today is *the agent remembering the rule* — which failed in practice this session.
- **Copilot CLI supports hooks** (docs: *Using hooks with GitHub Copilot CLI* + *GitHub Copilot hooks reference*). The **`preToolUse`** event fires **before each tool executes** and can return `permissionDecision: "deny"` (with a required `permissionDecisionReason` shown to the agent). Hook config for the CLI is loaded from `.github/hooks/*.json` (repo-level, committed) at **CLI start**.
- **Fail behavior matters (hooks reference).** Command `preToolUse` hooks are **fail-closed on crash/non-zero exit** — a buggy hook would deny *every* tool call — but **fail-open on timeout**. Therefore the decision script MUST be maximally robust: always exit `0`, and emit a deny decision only for the exact target tool; any parse error falls through to allow.
- **Tool name.** The memory-store tool's runtime name is `store_memory` (no Claude-format equivalent, so it keeps its runtime name; the payload field is `toolName` in camelCase form, `tool_name` in the VS Code `PreToolUse` form).
- **Node is guaranteed.** This is a Node ≥20 project, so a small ESM decision script parses the payload reliably without a fragile shell/`jq` dependency and is unit-testable; the helper itself **adds no dependency** and uses only Node built-ins (the repo does declare runtime deps such as `js-yaml`/`ajv`, so the "no deps" claim is about this helper, not the repo).
- **Composed file.** `.github/copilot-instructions.md` is a **composed** file; only the `copilot-instructions.harness` local block may be edited (the managed core is regenerated by `harness sync`). A local-block edit is self-host-scoped (consumers receive an empty block) and preserved across sync — consistent with the "self-host only" scope.
- **`/memory`** (a user-global toggle) can disable memory across sessions but is not repo-scoped or committed; a repo-level hook is the correct, versioned, per-repo enforcement.

## Decisions

| # | Decision | Choice | Rationale |
|---|---|---|---|
| C114-1 | Enforcement mechanism | A pre-tool-use command hook (the `PreToolUse` event, per C114-7) that **denies** the `store_memory` tool | Fires deterministically before every tool call, independent of agent recall; can deny with a redirect reason surfaced to the agent. |
| C114-2 | Decision-script implementation | A Node ESM helper (`.github/hooks/no-memory.mjs`) invoked via the hook's cross-platform `command` field | Node is guaranteed (Node ≥20 project); avoids fragile shell JSON parsing / `jq`; the helper **adds no dependency and uses only Node built-ins** (the repo itself declares runtime deps such as `js-yaml`/`ajv`, so "zero deps" applies to this helper, not the repo); unit-testable. |
| C114-3 | Fail-safe robustness | Script exits with code 0; emits the deny JSON **only** when `toolName`/`tool_name` === `store_memory`; any read/parse error falls through to allow (empty stdout). Write the decision with a **synchronous** stdout write and let the process end naturally (set `process.exitCode = 0`); **never** call `process.exit()` before stdout has flushed. | Command `preToolUse` is fail-closed on non-zero exit — a crash would block ALL tools; and `process.exit()` can truncate buffered stdout and silently drop the deny decision. Never crash; never deny anything but the exact target; never lose the decision. |
| C114-4 | Blocked-tool scope | `store_memory` **only** (not `vote_memory`) | The doctrine targets *persisting durable knowledge* to memory; `vote_memory` curates existing memories and creates no new knowledge. (See Q1.) |
| C114-5 | Proactive surfacing | Add **one imperative line** to the `copilot-instructions.harness` **local block** of `.github/copilot-instructions.md` | Always-loaded every session → routes agents to repo docs *before* they reach for the tool. Local block is self-host-scoped and sync-preserved. |
| C114-6 | Consumer propagation | **Out of scope** (self-host only); no `template/` copy | Per @henrik-me's selection (option 1). A `template/` copy is a separate design decision (the doctrine is arguably generic) — recorded as a follow-up (Q2). |
| C114-7 | Hook event form + matcher | Register the hook on the **PascalCase `PreToolUse`** event with `"matcher": "store_memory"` — the hooks reference documents matcher semantics for the PascalCase form (anchored `^(?:store_memory)$` against the runtime tool name, which `store_memory` keeps as it has no Claude alias). Keep the in-script exact-name check (on `tool_name`/`toolName`) as the authoritative defense in depth. | The docs explicitly confirm matcher support for `PreToolUse` / `permissionRequest` (not for camelCase `preToolUse`), so PascalCase is the evidence-grounded definitive choice. Matcher scoping means a Node/script **launch** failure (fail-closed) can only ever deny `store_memory` — the desired outcome — never another tool. The in-script check stays authoritative + unit-testable. |

## Deliverables

1. **`.github/hooks/no-memory.json`** — hook config registering a **`PreToolUse`** command hook **scoped by `"matcher": "store_memory"`** (per C114-7): `{ "version": 1, "hooks": { "PreToolUse": [ { "matcher": "store_memory", "type": "command", "command": "node .github/hooks/no-memory.mjs", "cwd": ".", "timeoutSec": 10 } ] } }` (LF, no BOM).
2. **`.github/hooks/no-memory.mjs`** — ESM decision script: read the payload from stdin **to completion**, parse JSON, and when `toolName`/`tool_name` === `store_memory` emit **exactly one** single-line JSON object `{"permissionDecision":"deny","permissionDecisionReason":"…"}` whose reason points at `LEARNINGS.md` / `INSTRUCTIONS.md` §45–61 and names the hook; for any other tool or on any parse/read error emit nothing. Write with a **synchronous** stdout write and end with exit code 0 **without** calling `process.exit()` before stdout flushes (set `process.exitCode = 0`), so the deny decision is never truncated (per C114-3).
3. **`tests/cs114-no-memory-hook.test.mjs`** — unit test (min. 4 cases): (a) `tool_name: "store_memory"` payload (the PascalCase `PreToolUse` shape) → stdout parses to exactly one object with `permissionDecision: "deny"` and a non-empty reason; (b) `toolName: "store_memory"` (camelCase shape) → same deny (the script handles both fields); (c) a non-memory tool (e.g. `bash`) → empty stdout (allow) + exit 0; (d) malformed / empty stdin → empty stdout + exit 0 (fail-open). Exercise the script via the **configured command** (`node .github/hooks/no-memory.mjs`) from repository-root cwd. Also assert `.github/hooks/no-memory.json` is valid JSON whose `PreToolUse` entry has the matcher (`store_memory`), `command`, `cwd`, and `timeoutSec`. All scratch under `os.tmpdir()` — never under REPO_ROOT.
4. **`.github/copilot-instructions.md`** — add one line to the `copilot-instructions.harness` local block: agent memory is disabled in this repo; durable knowledge goes to `LEARNINGS.md` / the relevant process doc; cross-link INSTRUCTIONS.md §45–61 and the `.github/hooks/no-memory.json` enforcement. (Managed core untouched; local-block edit only.)
5. **Verification:** `harness lint` 0-fail; `node --test tests/*.test.mjs` green (incl. the new test); `harness sync --mode=check` reports **no drift** (the local-block edit must not alter the composed managed core). After a **CLI restart** (hooks load at start), manually verify both that a `store_memory` attempt is **denied** and that a benign tool (e.g. a read) still **executes** — recorded at close-out.

## Risks + open questions

| # | Risk / what breaks | Mitigation |
|---|---|---|
| R1 | A buggy or launch-failing `preToolUse` hook is **fail-closed on crash/non-zero exit** and would block *all* tool calls. | C114-7 matcher scopes the hook to `store_memory` so a launch failure can only ever deny the target; C114-3 the script ends with exit 0, denies only the exact match, and writes synchronously (no `process.exit()` truncation); the fail-open unit test covers malformed input. |
| R2 | The hook only loads at **CLI start**, so an agent mid-session won't see it until restart. | Documented limitation (not fixable in-band); the instruction-line (C114-5) covers the in-session proactive path. |
| R3 | `.github/hooks/*.json` / `*.mjs` must pass `check-text-encoding` (LF/no-BOM) and must not trip other linters. | Author with LF/no-BOM; run `harness lint` before PR; verify no linter claims the new paths. |
| R4 | The local-block edit to a **composed** file accidentally drifts the managed core. | C114-5 / G114-2: edit only inside the `copilot-instructions.harness` markers; `sync --mode=check` clean is an exit gate. |
| Q1 | Should `vote_memory` also be denied? | Default **no** (curation ≠ knowledge storage); flip only on @henrik-me's confirmation (G114-1). |
